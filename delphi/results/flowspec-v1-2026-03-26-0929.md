# FlowSpec v1 Design Review — Final Recommendation

**Date:** 2026-03-26
**Process:** Delphi (2 workers, 1 judge, 2 critics)
**Subject:** Critical evaluation and improvement of FlowSpec DSL

## Contested Decisions (Resolved)

### 1. `->` vs `as` — Keep `->`

**Decision:** Keep `->`. Critic 2's `"Rewrite this as a haiku" as poem` example is devastating. PMs write prompts full of "as" — "act as an expert", "format as a table", "rewrite as bullet points". The `->` is a flowchart symbol PMs already use on whiteboards. You learn it once in 5 seconds and it never collides with English.

### 2. Semantic Conditions — Ship `means`, but with last-line classification

**Decision:** Keep the keyword `means`, but implement via last-line classification (not VERDICT prefixes).

Both critics landed real punches on naive `means` implementation:
- Prompt contamination degrades bot output
- Can't rewrite sub-workflow prompts
- PM loses ability to define category boundaries
- VERDICT tags leak into Slack-visible responses

**Refined mechanism:**

```
ask @clive "Review PR {pr_number}" -> review
if {review} means "security issue"
  ...
otherwise if {review} means "style issue"
  ...
```

The compiler:
1. Appends to the prompt: *"On your final line, classify your response as exactly one of: SECURITY ISSUE, STYLE ISSUE, OTHER"*
2. At runtime, extracts the last line for condition matching, **strips it** before storing the variable
3. The Slack-visible response also gets the last line stripped by the bridge (one bridge change)

No VERDICT prefix. The classification is a plain English label on the last line. Simpler to parse, nothing to leak.

**For sub-workflow outputs** where the compiler can't rewrite the prompt: emit a compile error. *"Variable {result} comes from `run` — use `contains` or `equals` instead of `means`."*

**Budget 200-300 lines** of compiler code (two-pass compilation required), not the originally estimated 100.

### 3. Approval — Keep `pause for approval`, fix `on reject`

**Decision:** Keep `pause for approval` + `on reject`. The `wait for approval -> decision` approach makes the common case worse.

Current (good for 80% case):
```
pause for approval with message "Ready to merge?"
ask @clive "Merge the PR"
```

Fix the magic `{feedback}` variable with explicit capture:
```
pause for approval with message "Review this RFC: {rfc}"
  on reject -> feedback
    ask @betty "Revise based on: {feedback}" -> rfc
```

### 4. Race syntax — Use `race`

**Decision:** Both `take the first` (ambiguous) and `fastest wins` (implies quality judgment) are flawed. `race` is short, maps directly to `Promise.race`, and PMs understand it:

```
race
  ask @betty "Find it in logs" -> finding
  ask @clive "Find it in code" -> finding
```

Keep `at the same time` for fan-out/fan-in. Use `race` for first-to-finish.

### 5. `collect` — Explicit target variable

**Decision:** Implicit "last output" is fragile. Make it explicit:

```
for each bug in {bugs}, collect {fix} as all_fixes
  ask @clive "Investigate {bug}" -> analysis
  ask @betty "Write fix: {analysis}" -> fix

ask @roger "Summarize all fixes: {all_fixes}"
```

### 6. Recursive `run` — Breadth guard, not just depth

**Decision:** Critic 1's exponential breadth explosion example is the real danger. A depth-50 limit with branching factor 4 = catastrophic cost. Change to:

```
run "Deep Research" with topic = {question}, at most 50 total
```

Where `at most N total` limits cumulative child workflow executions across the entire recursion tree. Compiler default: 50 total. Counter passed as workflow input and decremented at each `run` site.

## Unaddressed Issues (New from Critics)

### 7. Workflow-level timeout

```
workflow "Fix All Crash Bugs"
  timeout: 2 hours
  inputs: ...
```

Compiles to `workflowExecutionTimeout`. Trivial, high value.

### 8. Variable scoping

Compiler warns on conditionally-defined variables used outside the condition block. Undefined = empty string at runtime. No new `set` primitive needed.

### 9. Parallel error handling

Default to `Promise.allSettled` (best-effort), not `Promise.all`. Failed branches produce empty variables. Matches `for each` best-effort default. Update the compilation table.

### 10. List parsing

"Lightweight regex" won't cut it. Build a robust extraction function (~50 lines) that handles numbered lists, bullets, markdown, preamble/postamble. The compiler already appends "one item per line" — the parser handles the rest.

### 11. Pool references in parallel blocks

Allow `@coders` in `at the same time`. Enforce distinct *assignments* at runtime, not distinct *references* at parse time.

### 12. Indentation documentation

State explicitly: indentation is significant, 2-space blocks, no `end` keywords.

## Revised Primitive Table (Final)

| # | Primitive | Purpose | Temporal Mapping |
|---|-----------|---------|-----------------|
| 1 | `ask @bot "..." -> name` | Dispatch prompt, capture output | `executeActivity(dispatchToBot)` |
| 2 | `send @target "..."` | Fire-and-forget message | Fire-and-forget activity |
| 3 | `at the same time` | Parallel (all finish) | `Promise.allSettled()` |
| 4 | `race` | First to finish wins | `Promise.race()` + cancel scope |
| 5 | `for each X in {list}` | List iteration | `for...of` + continueAsNew |
| 6 | `repeat until` | Convergence loop | Bounded `while` + continueAsNew |
| 7 | `if / otherwise` | Conditional | `if/else` |
| 8 | `pause for approval` | Human gate | Signal wait |
| 9 | `run "Workflow"` | Sub-workflow (recursive OK) | `executeChild()` |
| 10 | `stop` | Early exit | `return` |

**Condition operators:** `contains`, `equals`, `is above`, `is below`, `is empty`, `is not empty`, `means` (compiler-injected last-line classification)

**Modifiers on `ask`:** `within <duration>`, `retry N times`, `if it fails`, `if it times out`, `(new session)`

**Loop modifiers:** `N at a time`, `stop on failure`, `collect {var} as name`, `if it never converges`

**Approval modifier:** `on reject -> feedback`

**Workflow-level:** `timeout: <duration>`

**Recursion guard:** `at most N total` (default 50)

## Key Syntax Changes Summary

| Original | Final | Rationale |
|---|---|---|
| `->` | **Keep `->`** | `as` collides with English in prompts |
| `notify` | `send` | Natural |
| `call` | `run` | Natural |
| `at the same time, take the first` | `race` | Short, unambiguous, maps to Temporal |
| Manual VERDICT in prompts | `means` with last-line classification | PM writes intent, compiler handles mechanism |
| `pause for approval` / magic `{feedback}` | `on reject -> feedback` | Explicit capture, no magic variables |
| No accumulation | `collect {var} as name` | Fills critical gap |
| No workflow timeout | `timeout: <duration>` | Safety net for long-running workflows |
| Depth guard on recursion | `at most N total` | Prevents breadth explosion |
| `Promise.all` for parallel | `Promise.allSettled` | Best-effort default matches `for each` |

## Feasibility Notes

### Highest-risk items (budget extra time):
1. **`means` compiler** — Two-pass compilation, data flow analysis across branches, last-line stripping in both compiler and bridge. 200-300 LOC.
2. **`race` + cancellation** — `CancellationScope` nesting in Temporal is tricky. Bridge must handle abort signals for in-progress bots.
3. **Recursive `run` with breadth guard** — Counter serialization across child workflows + `continueAsNew` variable forwarding.

### Lowest-risk items (ship first):
1. Renames (`send`, `run`, `stop`)
2. `race` keyword (parser change, codegen reuses existing `Promise.race` pattern)
3. `timeout:` header (one line of codegen)
4. `collect {var} as name` (~10 lines of codegen)

### `/cc check` must be excellent
PMs will draft workflows in Slack, Google Docs, Notion — all of which mangle whitespace. Error messages must be PM-friendly: *"Line 7: this line should be indented inside the 'if' block on line 5."*
